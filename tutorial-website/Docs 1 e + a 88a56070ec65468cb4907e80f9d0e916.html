1 e + a is a language designed for writing drum sheet music.

time: 4/4
division: 1/16
tempo: 120
title: example

pattern sixteenth_notes: [1 e + a | 2  e + a  | 3 e + a  | 4  e + a  |]

bar mybar:
  hh: [ sixteenth_notes ]
  sn: [   | 2 |   | 4 |]
  bd: [ 1 |   | 3 |   |]

render: { mybar }

Getting started

Running 1 e + a with an interactive editor and viewer

(Users shouldn't have to do this in the future, but for now you still have to clone the repo and run both the server and the client locally)

1. Clone the Github repository from

git clone https://github.com/williams-cs/cs334_project-stavaser.git
  1. Run the server which is inside server folder:
cd server
dotnet build
dotnet run
  1. Run the website which is inside the webapp folder:
cd ..
cd webapp
npm install
npm start
  1. Done! By default, the React project will start on http://localhost:3000/. You can start writing programs in the editor and the output will be rendered every time you hit save (cmd + s or ctrl + s if you‚Äôre on Windows).

Running 1 e + a from the command line

To run programs written in 1 e + a on your machine, download the Github repository from

git clone https://github.com/williams-cs/cs334_project-stavaser.git

Then when you‚Äôre inside the project folder, type

cd lang
dotnet build
dotnet run <filename>

<filename> is a text file that contains 1 e + a code.

The output of this command is a text file containing abc notation which you can then convert to pdf or svg by using abc‚Äôs software: https://abcnotation.com/software.

Or if you are at the solution level (the parent directory) you can run the project by typing

dotnet run --project lang/lang.fsproj <path_to_filename>

Tutorial: writing your first song in 1 e + a!

Follow these 3 simple steps to get familiar with 1 e + a syntax and get started writing your own programs.

Note that currently for the language to be able to render anything, it needs the settings explicitly written in at the top level. So, in all of these examples there‚Äôs the exact same ‚Äúheader‚Äù omitted for the sake of saving space:
time: 4/4
division: 1/16
tempo: 120
title: example

Step 1: creating a pattern

Note: Refer to the ‚ÄúNotes‚Äù and ‚ÄúPatterns‚Äù section in the Overview for a more detailed description of these data types.

A pattern is one measure of notes. Creating a pattern is very simple. All you have to do is use the keyword pattern and give it a name! Then after the semicolon (:), you can begin writing the pattern. The entire measure should be inside square brackets [] , and all beats in the measure should be separated by | :

time: 4/4
division: 1/16
tempo: 120
title: example

pattern eight_notes: [1 + | 2 + | 3 + | 4 + |]

...

We already have one pattern, but we‚Äôd like to create one more, which will be our bass drum pattern that we can reuse later on. So, let‚Äôs do that:

...

pattern eight_notes: [1 + | 2 + | 3 + | 4 + |]
pattern bass_drum_pattern: [1 |   | 3 |   |]

...

But oh no! We cannot render patterns alone because they needs assignments to instruments! The next example illustrates how you can do this and combine many patterns into a bar to render.

Step 2: creating a bar

Note: Refer to the ‚ÄúBars‚Äù section in the Overview for a more detailed description of this data type.

The process of creating bars is very similar creating patterns! All you have to do is use the keyword bar and give it a name, too. Then after the semicolon (:), you can begin writing the bar. For example,

bar my_cool_beat:
	...

Now we can assign patterns into various drums by writing <drum_name>: <pattern> inside the bar expression! Like this:

Note: For more information which drums are available to use in <drum_name>, refer to the ‚ÄúDrums‚Äù section in the Overview.
...
bar my_cool_beat:
	hh: [eight_notes]
	bd: [bass_drum_pattern]
...

But you are not restricted to always use pattern variable names to assign patterns to drums. You can directly write a pattern after <drum_name>: if you like! For example,

...
bar my_cool_beat:
	hh: [eight_notes]
  sn: [  | 2 |   | 4 |]
	bd: [bass_drum_pattern]
...

Cool! Now all you have to do is specify what to render after the render: keyword inside curly brackets {}:

render: {my_cool_beat}
Note: render accepts a list of bars or snippets (more on snippets in step 3) separated by commas. Read more on the ‚ÄúRendering‚Äù section in the Overview.

And that‚Äôs it, you‚Äôve created your very first bar, yaay! üéâ

The full example:

time : 4/4
division: 1/16
tempo: 120
title: example

pattern eight_notes: [1 + | 2 + | 3 + | 4 + |]
pattern bass_drum_pattern: [1 |   | 3 |   |]

bar my_cool_beat:
	hh: [eight_notes]
	sn: [  | 2 |   | 4 |]
	bd: [bass_drum_pattern]

render: {my_cool_beat}

Step 3: creating a snippet

Now that we have a bar, we can go even further and combine multiple bars into a snippet!

Note: Refer to the ‚ÄúSnippets‚Äù section in the Overview for a more detailed description of this data type.

Declare a snippet by writing

snippet my_song:
	...

We have 2 choices when creating a snippet:

  1. We can either repeat one or more bars N number of times by writing
    repeat N: {bar1, bar2, ...}
  1. Or change a repeated bar by specifying which repeated number to change like so
    change 10: my_bar (1,4,7){
	<drum_name>: <pattern>
}

So let‚Äôs do both!

...

snippet my_song:
	repeat 2: {my_cool_beat}

	change 16: my_cool_beat (every 4) {
	    cc: [1 | 2 | 3 | 4 |]
	    hh: [||||]
			sn: [1 e + a ||3 e + a||]
			t1: [|2 e + a|||]
			t2: [|||4 e + a|]
  }

render: {my_song}

Overview

This is a collection of basic syntax elements and their meanings followed by examples.

The order in which these elements appear in the program should be strictly followed:

  1. Settings
  1. Patterns
  1. Bars
  1. Snippets
  1. Render

... settings ...
... patterns ...
... bars ...
... snippets ...
render: {...}

Settings

There are 4 required settings to render the sheet music: time signature, division, tempo, and title. And the order must be strictly followed: first - time, second division, then tempo, and finally title.

Time signature

Format: time: <int>/<int>

Top number tells how many beats should be in one measure, and the bottom number tells what value of note should get the beat.

Note: The language currently supports only 4/4 time signature.

Division

Format: division: 1/<int>

Specifies the minimum value that notes can divide into.

Note: The language currently supports only 1/2, 1/4, 1/8, and 1/16 division of beats.

Tempo

Format: tempo: <int>

Specifies the speed of the music to be played.

Title

Format: title: <string>

Specifies the title of the sheet music.

Example

time : 4/4
division: 1/16
tempo: 120
title: example

Drums

Below is an example of a basic drum kit and their corresponding abbreviations used to write programs.

cc = Crash cymbal
hh = Hi-hat
rd = Ride cymbal
sn = Snare drum
t1 = High tom
t2 = Mid tom
ft = Floor tom
bd = Bass drum

More drums and techniques will be added in the future!

Rests

Rest is a special case of a drum. Think of it as an extra invisible instrument:

r = Rest

Rests behave exactly like any other drum instrument, thus they should be added only when the first note of a beat does not exist.

Example:

bar bar_with_rest:
	sn: [ + a | 2 | e + a |   |]
  hh: [ + a | 2 | e + a |   |]
  r:  [ 1   |   | 3     | 4 |]

Notes

Meaning

Notes in 1 e + a are written, well‚Ä¶ using the ‚Äú1 e and a‚Äù counting system:

For example, in 4/4 time and 1/16 division

  • ‚Äú1 2 3 4‚Äù means 4 quarter notes
  • ‚Äú1 and, 2 and, 3 and, 4 and‚Äù means 8 eighth notes
  • ‚Äú1 e and a, 2 e and a, 3 e and a, 4 e and a‚Äù means 16 sixteenth notes

Syntax

A sequence of notes in one measure is placed inside square brackets [] and separated by | after EACH beat:

[1 | 2 | 3 | 4 |]
Note: it is important that you put | after each beat even if there‚Äôs no note (meaning a rest) in the beat because it might be ambiguous which beat a note belongs to:
[1 + 2 + 4 a]
[1 + | 2 + |  | 4 a |]
General rule of thumb is that you should have as many |'s as the top number of the time signature.

Example

[1 |  | 3 |  |]

[1 ||||]

[1 + | 2 e a | + a | 4 e + a |]

Patterns

Meaning

A pattern is a data type that contains one measure of notes as defined by the time signature.

Syntax

pattern <pattername>: <notes>
  • <pattername> is a string denoting the pattern‚Äôs name
    • Allowed characters: any letters from the English alphabet and underscores _
  • <notes> is one measure of notes as defined in section ‚ÄúNotes‚Äù.

Example

pattern mypattern: [ 1 e + a | 2 e + a | 3  e + a | 4  e + a |]

Bars

Meaning

A bar is a data type that contains one or more pattern variable names or notes assigned to drum instruments.

Note: For more information which drum instruments are available to use, refer to the ‚ÄúDrums‚Äù section in the Overview.

Syntax

bar <barname>: 
	<drum>: [<pattern>]
  • <barname> is a string denoting the pattern‚Äôs name
    • Allowed characters: any letters from the English alphabet and underscores _
  • <drum> is a drum instrument
  • <pattern> can either be a pattern variable name or directly <notes>

Example

bar my_bar: 
	hh: [my_pattern]
	sn: [1|2|3|4|]

Snippets

Meaning

A snippet is a data type that contains one or more combinations of the two possible instructions:

  1. Repeat instruction: repeats one or more specified bar N times
  1. Change instruction: repeats ONE specified bar N times, but also changes K-th repetitions of the given bar.

Syntax

snippet <snippetname>: 
	repeat <N>: {<barname>, <barname>, ...}
	change <N>: <barname> (<option>) {
		<drum>: <pattern>
	}
  • <snippetname> is a string denoting the snippet‚Äôs name
    • Allowed characters: any letters from the English alphabet and underscores _
  • <N> is the number of times to repeat a bar
  • <barname> is a bar variable name to repeat/change
  • <option> is a repeat option which can have two forms:
    • A list of numbers followed by commas: 1, 2, 3, ‚Ä¶
    • A number after the keyword every <num>

Example

snippet my_snippet:

	repeat 2: { my_intro_one, my_intro_two }
	repeat 3: { something }

	change 16: my_beat (every 4) {
			t1: [1 | 2 | 3 | 4 |]
			t2: [ some_pattern ]
  }

Rendering

render keyword will render (or in other words, evaluate) the variables specified inside curly brackets {} after the semicolon :. The render keyword must always come the very last in the program.

Accepted data types are:

  • bar
  • snippet

Example:

...

render: {<barname>, <snippetname>, ...}
